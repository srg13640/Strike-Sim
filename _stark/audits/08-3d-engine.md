# Audit: 3D/Geo engine

Auditor surface: engine.js (EngineModule), stage.js (StageModule), and the boot/fallback
logic in StrikeSim2040.html (webglAvailable, tryInitGraph3D, handleGraph3DFailure,
degradeToMap, retry3D, geo toggles, colorFromTeam, refreshGraph/applyHighlight). Cannot run
WebGL headless; findings are from static reasoning about the lifecycle and the 3d-force-graph /
Three.js r128 contract.

Overall: the fallback path is solid and calm, and the boot capability gate is sensible. The
biggest real gaps are (1) a detection probe that asks for a context the real renderer never
requests, so it can return true and then the renderer still fails, and (2) the 3D scene being
visually generic — default sphere sizing, no in-scene labels, no tactical symbology — so "once
working, does it look credible" is the weakest answer. Context-loss recovery has a one-shot
attach bug across retries.

---

## [P0] webglAvailable() probe does not match what 3d-force-graph actually requests
Where: StrikeSim2040.html `webglAvailable()` (~3279-3288); compare engine.js `create()` rendererConfig (~352-358).
Problem: The probe calls `getContext('webgl2', opts) || getContext('webgl', opts) || getContext('experimental-webgl', opts)` with only `{failIfMajorPerformanceCaveat:false, powerPreference:'default'}`. 3d-force-graph / three.js r128's WebGLRenderer requests a context with a *different* attribute set (antialias:true, alpha:false, plus three's own defaults like `preserveDrawingBuffer`, `stencil`, `depth`). On constrained drivers (SwiftShader, remote/virtual displays, some headless GPUs) a bare context can succeed while the renderer's attribute combination, or three's post-create capability checks (float textures, required extensions, max varyings), fails — so the probe returns true, `tryInitGraph3D()` runs, and the renderer throws anyway. The reverse is rarer but possible: probe requests webgl2 first, succeeds, yet three on r128 uses webgl1 — masking nothing here, but it means the probe's "true" is not the renderer's "true." Net: `webglAvailable()` is an optimistic gate, not a faithful predictor, which undermines every branch that trusts it (the calm-notice selection in handleGraph3DFailure, and the retry3D no-op guard).
Fix: Make the probe construct the context with the *same* attributes the engine uses (mirror engine.js's default rendererConfig: antialias:true, alpha:false, powerPreference:'default', failIfMajorPerformanceCaveat:false, preserveDrawingBuffer:false), and additionally assert the few capabilities three needs (e.g. `gl.getParameter(gl.MAX_TEXTURE_SIZE) > 0`, and that `getContextAttributes()` returned non-null). Keep the webgl2-or-webgl fallback. Better still, export a single shared `probeContext(config)` used by both the probe and `create()` so they can never drift. Treat the probe as advisory: tryInitGraph3D's try/catch is the real source of truth — which it already is, so mainly tighten the *messaging* branches that read the probe.
Effort: M

## [P0] Context-loss handler attaches only once; survives the first GPU drop but not a post-retry one
Where: stage.js `apply()` (~66) and `attachWebGL()` (~81-96); the `webglAttached` module flag (~25).
Problem: `webglAttached` is a module-level boolean set true the first time a renderer exists, and never reset. When the operator hits "Retry 3D", retry3D calls `EngineModule.dispose()` (which `forceContextLoss()`es and nulls the old instance) then `EngineModule.create()` builds a *brand-new* renderer with a *brand-new* canvas/domElement. Because `webglAttached` is still true, `apply()` never calls `attachWebGL(inst)` again, so the new canvas has NO `webglcontextlost`/`webglcontextrestored` listeners. After a successful retry, a subsequent GPU context drop (fullscreen transition, resize storm) will hit the canvas with no `preventDefault()` — the browser will not fire `restored`, and the 3D view dies permanently with no recovery, exactly the failure stage.js exists to prevent. The same applies to the geo `dispose()`/recreate path.
Fix: Reset `webglAttached = false` whenever the instance identity changes. Simplest robust approach: in `apply()`, track the last canvas element (`lastCanvas`) and re-attach whenever `inst.renderer().domElement !== lastCanvas`. Or expose an `EngineModule` callback/event on create that calls `StageModule` to re-arm. Avoid double-attaching to the same canvas (guard by canvas identity, not a global bool).
Effort: S

## [P1] 3D scene uses default node sizing/labels/links — looks generic, not a credible COP
Where: engine.js `create()` (~359-377) sets only `nodeLabel`/`nodeColor`/click handlers; StrikeSim2040.html `graph3dOpts` (~3262-3275). No `nodeVal`/`nodeRelSize`/`nodeThreeObject`/`nodeResolution`/`linkWidth` defaults are ever set for the base view.
Problem: Every node renders as an identical default-size sphere regardless of importance/cascade weight (the app already computes `nodeValue(n) = importance * casc` in game.js but never feeds it to the graph). There are no in-scene labels — names appear only on hover tooltip (`nodeLabel`), so a screenshot or a glance shows anonymous colored dots. Default `nodeResolution` (8) gives faceted, low-poly spheres up close. Links use library defaults (thin, flat) except when a highlight mode is active. Against the polished Map view (2525 symbols, threat rings, strike arcs per the cinematic loop), the 3D view reads as a generic force-graph demo, not a command picture. This is the core "does 3D look credible" gap.
Fix: Set `nodeVal(n => nodeValue(n))` (or importance) and a sensible `nodeRelSize`, bump `nodeResolution` to ~12-16, and add optional in-scene labels via `nodeThreeObject` using a SpriteText/canvas-sprite (vendor three-spritetext or a small canvas-texture sprite) toggled by a "labels" control so dense scenes stay legible. Style base links (`linkWidth`, slight `linkOpacity`, optional `linkDirectionalParticles` for active strike/comm links) so relationships read at rest, not only when highlighted.
Effort: M

## [P1] Tactical symbology (symbols.js) is not carried into 3D — map and globe disagree visually
Where: symbols.js (`SymbolModule.svg(node)` ~260, `.divIcon` ~274) is used by map.js for Leaflet markers; engine.js never references SymbolModule.
Problem: In Map view a hostile fires battery shows a red 2525 frame with a fires glyph; switch to 3D (or Geo globe) and the same unit is a featureless red sphere. Affiliation, dimension (air/sea/ground), and function are all lost in 3D, so the two primary views present incompatible mental models of the same scenario — a real credibility and usability problem for a COP/wargaming tool. `SymbolModule.svg(node)` returns a self-contained SVG string, which is directly convertible to a sprite texture (serialize SVG -> data URL -> Image -> `THREE.Texture` -> `THREE.SpriteMaterial`), so the seam already exists.
Fix: Add a `nodeThreeObject` in engine.js (or via opts from the main script) that builds a `THREE.Sprite` from `SymbolModule.svg(node)` rasterized onto a canvas/texture, with a sensible billboard size and a small cache keyed by the symbol's SIDC-ish signature (team+domain+type+status bucket) so you are not rasterizing per-frame or per-node. Fall back to the current colored sphere if SymbolModule is absent or rasterization fails. This makes Geo mode in particular look like a genuine global COP. Gate behind the same labels/symbology toggle as the P1 above to keep dense graphs fast.
Effort: M

## [P1] Geo mode hard-codes globe radius (400) in layout but engine constant is 398 — and frame math assumes 400
Where: engine.js `EARTH_RADIUS = 398` (~34); `applyGeoLayout` pins nodes at `latLonToXYZ(lat, lon, 400 + alt)` (~299) and `latLonToXYZ` default radius is 400 (~137); `frameGeo` filters pinned nodes by `n.fy > -400` (~273) and the no-coord pool is parked at `fy = -500` (~307).
Problem: Three different "globe radius" magic numbers (398 mesh, 400 node pin, 400 default) are in play. Nodes sit 2 units above the mesh surface — fine for depth sorting, but it is incidental, not expressed. More importantly the `frameGeo` centroid filter `n.fy > -400` is a brittle proxy for "is this node on the globe vs in the no-coord pool at -500": any legitimately-pinned node near the south pole has `fy` approaching `-400` (radius*cos(phi) at high southern latitude → about -397), so a cluster of high-southern-latitude nodes can be *excluded* from the centroid (fy just above -400 still passes, but the margin is ~3 units and `alt` can push it the wrong way), and conversely the -500 pool is only safe because of an arbitrary 100-unit gap. The framing silently misbehaves for polar theaters.
Fix: Stop overloading the y-coordinate as a "pinned?" flag. Tag pooled nodes explicitly (e.g. `node.__geoPool = true`) in `applyGeoLayout` and have `frameGeo` test that flag instead of `fy > -400`. Unify the radius into the single `EARTH_RADIUS`-derived constant (pin at `EARTH_RADIUS + 2 + alt`) so mesh, pins, and frame math cannot drift.
Effort: S

## [P1] Geo at first load is mis-framed: onFirstSettle calls resetView() which calls frameGeo, but applyGeoLayout already framed — double-move; and frameBlueToRed never runs in geo
Where: graph3dOpts.onFirstSettle (~3268-3271): `if (geoModeEnabled||map||table) resetView(); else frameBlueToRed()`. resetView in geo path calls `EngineModule.frameGeo(1000)` (~2664). applyGeoLayout already calls `frameGeo()` (~317).
Problem: If the operator somehow lands in geo mode before first settle (or geo is toggled during the initial load window), the camera is framed twice in quick succession (applyGeoLayout's `frameGeo(800)` then onFirstSettle's `frameGeo(1000)`), producing a visible double-pan/jerk. More subtly, the marquee "Blue→Red opening shot" (`frameBlueToRed`) is the signature cinematic for this tool, and it is *unconditionally suppressed* whenever map/table/geo is the landing mode — which is now the common case on this machine (auto-routes to Map). So most sessions never see the opening shot at all; when the operator later hits Retry 3D, retry3D calls `frameBlueToRed(1200)` directly (good), but the *first* impression for a WebGL-capable machine that happens to default into a non-3D mode is no cinematic.
Fix: Make geo framing idempotent within a short window (skip a frameGeo if one fired <500ms ago), and reconsider the onFirstSettle policy so that an operator who *does* reach 3D gets the Blue→Red shot once even if they arrived via Retry (retry3D already handles this, so mainly ensure no competing frameGeo fires). Low-stakes but it is the engine's headline shot.
Effort: S

## [P2] retry3D / first-settle race: dispose() resets initialViewDone, but a stale onEngineStop from the disposed instance can fire onFirstSettle on the wrong instance
Where: engine.js `dispose()` sets `initialViewDone=false` (~453) and nulls `graphInstance`; `create()`'s `onEngineStop` closes over the module-level `graphInstance` and reads `graphInstance.graphData()` (~371-376).
Problem: `onEngineStop` references the module variable `graphInstance`, not the instance it was created for. During a dispose→create cycle (retry, or geo dispose), if the OLD engine fires a final `onEngineStop` after `create()` has reassigned `graphInstance` to the NEW instance, the callback reads the new instance's data and can fire `onFirstSettle` against a half-initialized new graph (or double-fire the opening shot). `_destructor()` should stop the old loop, but ForceGraph3D's engine-stop is debounced/async and this ordering is not guaranteed across the forceContextLoss path.
Fix: Capture the instance in a local inside `create()` (`const inst = ForceGraph3D(...)...`) and have `onEngineStop` operate on `inst`, bailing if `inst !== graphInstance` (stale). This makes the lifecycle robust to overlapping create/dispose.
Effort: S

## [P2] applyHighlight's direct material push sets BOTH color and emissive to the team color — washes out lighting, and never reverts emissive
Where: StrikeSim2040.html applyHighlight (~2723-2737): for each node `obj.material.color.set(col); obj.material.emissive.set(col)`.
Problem: Setting `emissive` equal to the full team color makes every node self-illuminated at full saturation, defeating the scene lighting and flattening depth (everything glows uniformly bright regardless of position/shading) — the opposite of the "credible from-orbit" look the geo lighting rig (key/fill/rim in engine.js) is trying to achieve. It also means dimmed/base nodes in highlight mode still emit their dim color at full strength rather than receding. And nothing ever sets emissive back to black when highlight mode clears — it is overwritten with the next color, so the scene can never return to lit (non-emissive) shading once this path runs.
Fix: Either drop the emissive push entirely (let `nodeColor` + lighting do the work) or set emissive to a heavily scaled-down color (e.g. multiply by ~0.15) and explicitly reset emissive to 0x000000 when not highlighting. Decide one shading model for nodes and apply it consistently in both base and highlight states.
Effort: S

## [P2] Geo no-coordinate nodes are scattered randomly each refresh — they jump between frames
Where: engine.js applyGeoLayout (~302-309): uncoordinated nodes get `Math.random()` angle/radius in the -500 pool every call.
Problem: `applyGeoLayout` runs on every `refreshGraph()` while geo is on (refreshGraph re-calls it ~2086-2088). Each call re-randomizes the pooled (no lat/lon) nodes' positions, so on every filter change, strike, or sim tick the uncoordinated units teleport to new spots in the ring below the globe — visually noisy and makes them un-trackable. Determinism matters for a wargaming tool (and the project already uses seeded RNG elsewhere).
Fix: Make the pool placement deterministic and stable — only assign a pool position if the node does not already have one (cache on the node, e.g. `node.__geoPoolPos`), or derive the angle from a hash of `node.id` so a given unit always lands in the same ring slot. Skip re-randomizing nodes that are already pooled.
Effort: S

## [P2] Console cleanliness: console.warn override is global and permanent; console.error in init paths still fires on the expected no-WebGL machine
Where: StrikeSim2040.html warn-suppressor IIFE (~1519-1525); tryInitGraph3D `console.error('3D init attempt N failed', err)` (~3305); degradeToMap `console.error(...)` (~3348,3352).
Problem: (1) The `console.warn` monkey-patch is installed permanently and globally and swallows *any* warning whose first arg exactly equals the three.js multi-instance string — fine today, but it is a process-wide override with no teardown, and an exact-string match is brittle (a version bump changes the text and the suppression silently stops working). (2) On the target machine (no WebGL), the boot path takes the clean `webglAvailable()===false` branch and does NOT spam — good — but if the probe ever returns a false-positive (see P0), tryInitGraph3D logs `console.error` for each of the 2 profiles before degrading, putting two red errors in the operator console for an environmental limitation the team has deliberately made calm everywhere else. The user-facing toasts are calm; the console is not.
Fix: Downgrade the per-profile failure logs to `console.debug` (or gate behind a `?debug=1`), keeping a single concise `console.warn` summary if all profiles fail. Make the warn-suppressor match by substring/regex and consider restoring the original `console.warn` after the graph has initialized. The P0 probe fix is the real cure since it stops the false-positive that triggers the error spam.
Effort: S

## [P2] Earth texture upgrade path and atmosphere/starfield rebuild assumptions on retry
Where: engine.js `ensureEarthSphere` HEAD-probe + TextureLoader (~217-243); `dispose()` resets `earthMesh=null`, `starField=null`, `earthLightingDone=false` (~454-456) but lighting is added to the shared `scene`, not the `earthMesh` group.
Problem: The lights (ambient/key/fill/rim) are added directly to `scene` (~190-193), guarded by `earthLightingDone`. On `dispose()` the flag is reset to false but the OLD lights were on the OLD instance's scene, which is gone — fine. BUT if geo mode is toggled off and on within the SAME instance, `ensureEarthSphere` early-returns on `if (earthMesh) return` so lighting is added exactly once — correct. The real latent issue: the texture HEAD-probe (`fetch(EARTH_TEX_URL, {method:'HEAD'})`) runs every time `ensureEarthSphere` builds (i.e. once per instance); the asset is currently absent (confirmed: no assets/earth-blue-marble-2048.png in the tree), so the procedural globe is always used — meaning the "photo globe + opaque occlusion + hidden graticule" path is dead code in the field and the semi-transparent dark globe (opacity 0.62) is what ships. Far-side nodes show *through* the planet, which can read as confusing depth. Also `cache:'force-cache'` on a HEAD that 404s still issues a network request per instance (cheap, but it is a request on an offline-hardened tool).
Fix: Either bundle the Blue Marble texture so the credible photo-globe path actually runs, or accept the procedural globe and bump its opacity toward opaque (and keep the graticule) so far-side nodes are properly occluded and the globe reads solid. Guard the HEAD probe behind a build-time flag or a one-time check so an offline tool with no texture makes zero network calls. Confirm the documented "no 404 / no request" claim holds (HEAD to a missing local file under file:// or a static server still hits the network/filesystem).
Effort: S
