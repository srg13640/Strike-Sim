## 1. Context

- **Project:** Strike Sim (`/Users/sethgilleland/Library/Mobile Documents/com~apple~CloudDocs/01_Active_Projects/Strike Sim`)
- **Author:** Seth Gilleland
- **Why a loop:** The turn-based War Game is broken in a specific way: Red dominates Blue from turn 2 onward, and Blue effectively can only "fill out" turn 1. Investigation shows the wargame engine ignores the per-node resource fields the scenario JSONs already carry (`resourceGenByType`: 0 references in `game.js`; `subsystem`: 0 references; `importance` / `cascScore`: barely used). So the fight collapses to a raw node-count contest (Red 150 vs Blue 90), unrelated to the resourced posture the scenarios were designed to express. **The fix is structural plumbing, not a balance dial.** A loop is the right tool because each candidate change needs to be verified against (a) the perturbation tests proving the tool ties to itself, and (b) the 100-match auto-tournament on the bundled scenarios. Hand-tuning won't catch regressions across both tests.

## 2. Artifact

- **Path:** `Strike Sim/game.js`
- **Type:** code
- **Inputs (read-only):**
  - `grokblue90.json` — Blue 90-node scenario, as-shipped
  - `grok150red.json` — Red 150-node scenario, as-shipped

The loop touches **only `game.js`** and may only ADD references to scenario fields — not change the scenario file schema, not change the public API of `GameModule`.

## 3. Signal

The loop accepts a change only if **both** parts move in the right direction and neither regresses.

### Part A — "the tool ties to itself" (perturbation tests)

Run 4 perturbed copies of the scenarios and confirm the wargame responds in the expected direction.

| # | Perturbation                                                  | Expected wargame response                                                                  |
| - | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| 1 | Double `resourceGenByType.kinetic` on every Blue node          | Blue's avg kinetic-strike orders per turn ≥ **1.8×** the baseline run                       |
| 2 | Zero out `resourceGenByType.kinetic` on every Red node         | Red's avg kinetic-strike orders per turn ≤ **20%** of baseline                              |
| 3 | Set every Red node's `subsystem` to "Logistics"                | Red's avg Firepower-Strike orders per turn ≤ **20%** of baseline (logistics shouldn't strike) |
| 4 | Double `importance` on Blue's top 5 cascade nodes              | When those nodes are struck, the cascade damage to neighbors is measurably higher than baseline (≥ **1.3×**) |

A run passes Part A iff **all 4 perturbations produce the expected direction**.

### Part B — "default scenarios produce a fight"

Run **100 auto Blue-vs-Red matches** on the bundled scenarios (as-shipped, no perturbation), Red and Blue both controlled by the existing AI, fog off, 10-turn limit.

- **Blue win rate must be 35% – 55%** (a real fight, not a stomp either direction)
- **Average match length must be ≥ 6 turns** before someone wins or hits the limit

### Metric direction

Both parts must pass per iteration. Primary metric to minimize:

```
penalty = (failed_perturbations × 25)
       + max(0, |blue_win_rate − 0.45| − 0.10) × 100   # distance from balanced
       + max(0, 6 − avg_turns_to_decision) × 5          # too-short matches
```

- **Baseline penalty:** unknown until first measurement run; expectation is ~75–150 (all 4 perturbations probably fail today, Blue wins ~0–10%, matches end ~turn 2–3).
- **Target:** penalty ≤ **5** (all perturbations pass, win rate within 35–55%, avg turns ≥ 6).

### How it's measured

A new file `Strike Sim/tools/wargame-loop-eval.js` runs both parts and prints a JSON line:

```json
{"penalty": 12.3, "blue_win_rate": 0.41, "avg_turns": 7.2,
 "perturbations": {"1": true, "2": true, "3": false, "4": true}}
```

Invocation: `node tools/wargame-loop-eval.js --matches 100 --seed-base 42`

### Correctness gate (must stay true between iterations)

- The wargame still launches from the existing button (`#wg-launch`)
- Both bundled scenarios still load without console errors
- The campaign planner's "launch War Game" handoff still works (no thrown errors on transition)
- The `GameModule` public API surface is unchanged (same exported function names and signatures)
- Existing scenario JSON files (`grokblue90.json`, `grok150red.json`) remain valid — no schema change required of any data file

- **Gate command:** `node tools/wargame-loop-gate.js` (returns exit 0 on pass)

## 4. Termination

- **Iteration budget:** **8**
- **Wall-clock:** ~60 min (each iteration = ~100 matches + 4 perturbation runs ≈ 5–7 min)
- **Plateau rule:** stop after **3** consecutive non-improving iterations
- **Early exit:** stop immediately if penalty ≤ 5 (target reached)

## 5. Loop choice

- **Primary:** `optimize-loop` (`code` mode)
- **Why:** Single artifact (`game.js`), explicit correctness gate (perturbations + smoke tests must pass), explicit metric to minimize (penalty). Hill-climb is right because the fix is structural — wire `resourceGenByType` and `subsystem` into the action budget and the AI order picker. A tournament loop would be overkill; an exploratory loop would over-explore when the direction is known. If after 8 iterations the structural plumbing is in but balance is off, swap to `tournament-autoresearch` for round-by-round balance competition.
- **Alternates:**
  - `tournament-autoresearch` — if after 8 iterations the perturbations pass but win rate is still outside 35–55%, swap to this for multi-candidate balance tuning.
  - `ml-autoresearch` — if the loop keeps gaming the penalty without making the perturbations pass, switch to this for diagnosis-first iteration.

## 6. Run mechanics

- **Run command (baseline & each iter):** `node tools/wargame-loop-eval.js --matches 100 --seed-base 42`
- **Gate command:** `node tools/wargame-loop-gate.js`
- **Ledger:** `Strike Sim/loop-runs/<YYYYMMDD-HHMM>/ledger.tsv` — one row per iteration: `iter#`, `penalty`, `blue_win_rate`, `avg_turns`, perturbations 1-4 pass/fail, gate pass/fail, accepted?, one-line summary of the change tried.
- **Determinism:** `--seed-base 42` makes both the per-match seeds and the perturbation seeds deterministic. Record the seed in the ledger header.

## 9. Acceptance — what success looks like

- **Definition of success:** All 4 perturbation tests pass (the tool ties to itself); Blue wins between 35% and 55% of 100 auto-matches on the bundled scenarios; average match length is ≥ 6 turns; the gate passes; and the `game.js` diff is *understandable* — it should read like "plumbed `resourceGenByType` into the action-budget calculation; routed `subsystem` into the AI order filter; weighted `importance`/`cascScore` into target selection." If the diff is unreadable spaghetti even though the metric moved, that's failure (see below).
- **Definition of failure (do not ship even if the metric moved):**
  - Diff includes a rewrite of `GameModule`'s public API or restructures the resolver in a way that changes the meaning of "turn"
  - Loop gamed the penalty by adding scenario-specific magic numbers instead of reading from `resourceGenByType` / `subsystem` (i.e. it hardcoded `if scenario==grokblue90 boost blue`)
  - Any perturbation passed because the loop *clamped* the response (e.g. trivially scaling outputs) rather than wiring the field in
  - Win rate moved by *handicapping Red* (e.g. Red's AI stops choosing strikes) rather than empowering Blue's resourced posture
- **Review method:** `git diff game.js` side-by-side with the ledger; replay iteration #N's eval command; play one human-vs-AI match in the app and confirm Blue's per-turn action menu now reflects the loaded resources; spot-check the perturbation runs by hand on one of the four scenarios.
